## SDLC との関係：なぜ段階的に設計するのか

[English](../../en-US/theory/sdlc.md) | [中文](../../zh-CN/theory/sdlc.md) | [日本語](../../ja-JP/theory/sdlc.md)

visual-spec は、SDLC の中で「後工程で発覚すると高コストになる不一致」を、早期のレビュー可能な成果物へ寄せる設計です。誤解や欠落は、要件/設計の段階で直すのが最も安く、実装・テスト・運用に入ってからだと連鎖的な手戻りになります。

### コマンド → SDLC のマッピング

- `/vspec:new`：Discovery / Requirements
  - ロール、用語、シナリオ、フロー、機能一覧、依存関係、未解決質問を整備し「共通言語」を作る
- `/vspec:detail`：Design
  - 機能を実装可能な粒度の spec に落とす（RBAC、データ権限、交互、バリデーション、ログ、通知、MQ、入出力、定期実行など）
- `/vspec:verify`：Validation
  - データモデルと実行可能プロトタイプで、視覚的に振る舞いをレビューできるようにする
- `/vspec:qc`：Quality Gate
  - 抜け・矛盾・テスト不能・追跡不能などをルールベースで検出する
- `/vspec:accept`：Acceptance
  - 重要シナリオを受入用ケースに落とし、検証可能な形にする
- `/vspec:impl`：Build Inputs
  - 選定スタックとリポジトリ規約に合わせた実装入力を出す（有効時）
- `/vspec:append-test`：Test Automation
  - 受入用ケースからテスト計画/骨格を生成し、運用定着のコストを下げる
- `/vspec:plan`：Planning
  - 不確実性が収束し、品質が通ってから見積・排期を行う

### なぜ「巨大な PRD 1 本」にしないのか

- 変更に強くする：段階分割で影響範囲を限定し、書き直しを減らす
- レビュー対象が違う：ビジネスはシナリオ/プロトタイプ、開発は詳細 spec/モデル、QA は受入/可測性
- 追跡可能にする：シナリオ → spec → プロトタイプ → 受入 → テストの連鎖を作りやすい
