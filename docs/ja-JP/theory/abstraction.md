## 抽象：承認フローを一枚の図で統一する

`visual-spec-flows.svg` は特定の業務だけを表す図ではありません。承認/回付型の多くの業務フローを同じ骨格に正規化するための“再利用可能な抽象”です。これにより、レビュー・見積・実装の合意形成が格段に楽になります。

![visual-spec flows](../../assets/ja-JP/visual-spec-flows.svg)

ここで示しているのは「承認/回付型フロー」の抽象例の 1 つに過ぎません。実際のシステムには他にも多くの抽象（例：リソーススケジューリング、課金/精算、与信/適格性、在庫/履行、照合と資金フロー、イベント駆動協調など）が存在し、本ページでは列挙しません。

重要なのは「先に抽象し、その後にプロンプトを設計する」ことです。抽象が“必ずカバーすべき次元と境界”を固定し、プロンプトはそれを安定して再現できる出力形式に落とします。方法論の支えがないまま場当たり的にプロンプトを書いても、同じレベルの網羅性・実装可能性・検証可能性を安定して得るのは難しくなります。

### なぜこの抽象が効くのか

承認系フローの共通部分（申請 → 承認 → 実行 → 後処理）はほぼ同じで、差分は主に次の点に集約されます。

- 承認チェーンの構成（段階、合議、条件分岐）
- 実行を止める制約（検証、上限、コンプライアンス、並行、冪等）
- 例外系の制御（取消/却下、緊急停止、緊急変更、衝突解決）

骨格を固定すると、分析の焦点が「この業務の差分は何か」に移り、毎回フローを描き直すコストが消えます。

### プロンプト設計（安定出力のためのチェックリスト化）

この図をプロンプトの出力チェック項目として固定します。

- 各ステップの対応表（役割、入力、出力、状態遷移）
- 制約リスト（検証、失敗時の扱い、冪等キー、並行制御）
- 制御パス（取消/却下/取り下げ/ロールバック/補償の条件と権限）
- 検証成果物（分岐と主要制約を網羅する受入ケース）

### 抽象後の共通特性（thinking-modes に基づく）

抽象化後、承認/回付型フローの多くは「安定した主経路 + 列挙可能な制御パス + チェックリスト化できる実行制約」という同じ構造になります。差分を漏れなく露出させるために、[thinking-modes.md](file:///Users/stephenwang/Documents/trae_projects/my_skills/docs/ja-JP/theory/thinking-modes.md) を併用します。

- 境界思考：スコープ/ロール/権限/時間/データ境界を明確化
- 対称思考：逆方向の流れと補償（撤回/ロールバック/冪等）を導出
- 制約思考：検証、状態機械、監査/コンプライアンス、並行制御を顕在化
- 多様性思考：多ロール/多シナリオ/多チャネルで抜け漏れを抑制

また、「承認」と「実行」は単一ステップに限らず、複数ステップのサブフローになり得ます。重要なのは、それらを同じ制御/制約/成果物の次元に揃えて扱うことです。
