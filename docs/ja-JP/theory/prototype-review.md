## なぜシナリオ一覧を HTML で出力し、プロトタイプと連動させるのか

[English](../../en-US/theory/prototype-review.md) | [中文](../../zh-CN/theory/prototype-review.md) | [日本語](../../ja-JP/theory/prototype-review.md)

レビューが遅くなる主因は、次の 3 つに集約されがちです：

- 見ているものが違う：PRD/プロトタイプ/口頭説明が混在し、結論が再利用できない
- 指摘が紐付かない：どのシナリオ/ページ/操作の話かが曖昧で、追跡できない
- 検証が高コスト：文書だけでは振る舞いを検証しづらく、実装後に問題が顕在化する

visual-spec は「シナリオ一覧」を HTML のレビュー入口として出力し、実行可能プロトタイプと連動させることで、この問題を抑えます。

### なぜ HTML（Markdown だけではなく）

- レビュー導線に強い：クリック遷移、目次、クロスリンク、素早い定位
- 非技術者に優しい：ブラウザで開けば同じ見え方になりやすい
- レビューをタスク化できる：一覧として一つずつ確認しやすい

### なぜ「シナリオ ↔ プロトタイプ」連動

- シナリオは最小の有効単位：利用者は「流れが正しいか」で判断する
- ロール差分を直接検証できる：役割別ダッシュボードや権限制御を視覚的に確認できる
- データ口径の衝突を早期に露出できる：状態・項目・金額/時間ルール・端数処理など
- 受入に転用できる：レビューで確定したシナリオは受入用ケースに変換しやすい

### レビューが楽になる理由

- 指摘をアンカーできる：シナリオ/ページ/操作へ紐付け、追跡可能になる
- 想像コストを下げる：動くプロトタイプで理解差分を縮める
- フィードバック回路が短い：refine で修正し、下流成果物へ同期更新しやすい
