## 分析思维：把“需求分析”拆成可复用的模块

[English](../../en-US/theory/thinking-framework.md) | [中文](../../zh-CN/theory/thinking-framework.md) | [日本語](../../ja-JP/theory/thinking-framework.md)

需求分析不是一套固定模板，而是一组可复用的思维模块。visual-spec 把这些模块显式化，并在不同命令阶段按优先级调用，以保证“先收敛口径，再推进落地”。

### 模块 1：目标与边界（Goal/Scope）

- 明确“要解决什么问题、给谁解决、做到什么程度”
- 输出可验证的成功标准与非目标（Out of scope）

### 模块 2：角色与权限（Roles/RBAC）

- 以“角色 → 可见范围 → 可操作行为”建立权限基线
- 把权限下沉到页面区域/控件与数据行列范围，避免只做“菜单级权限”

### 模块 3：场景与流程（Scenarios/Flows）

- 用“触发 → 主路径 → 分支/异常 → 回滚/补偿 → 完成条件”描述场景
- 用流程图/状态机保证覆盖“可走通”与“可恢复”

### 模块 4：数据与口径（Data/Definitions）

- 提前统一字段含义、计算口径、状态含义、时间/金额/数量的展示与取整/精度规则
- 为模型与接口准备可复用的数据定义

### 模块 5：规则与约束（Rules/Constraints）

- 把隐性规则显性化：校验、幂等、容错、对账、审计、留痕、合规、权限叠加
- 把约束沉淀为“可检查项”，为 QC/验收提供依据

### 模块 6：可测性与可追踪性（Testability/Traceability）

- 让每个功能点都能映射到：输入/输出/预期结果/异常处理/日志与审计点
- 用验收用例与测试骨架承接评审结论，减少重复写用例成本

### 模块 7：评审与迭代（Review/Refine）

- 用可运行原型 + 场景评审入口缩短反馈回路
- 用 refine 驱动下游产物同步更新，避免“文档改了但原型/细节/实现没跟上”
