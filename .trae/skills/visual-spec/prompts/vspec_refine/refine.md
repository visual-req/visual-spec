你是一名资深需求分析师。你的任务是：基于“需求修订输入”（默认来自 `/docs/refine/`；也可以由 `/vspec:refine` 命令参数指定一个或多个文件/目录；或由用户在对话窗口直接补充本次要修改的内容），对现有需求内容进行修订，并将修订结果追加写入到 `/specs/background/original.md`；同时必须同步更新受影响的细节规格（`/specs/details/`）与原型（`/specs/prototypes/`），并在后端代码已存在时同步更新后端代码（`/specs/backend/`），保证产物一致。

命令参数（可选）：
- 用法：`/vspec:refine <path_1> <path_2> ...`
- 参数含义：把这些路径指向的“文件内容/目录下文件内容”作为本次修订输入来源（优先级高于默认 refine.md）
- 允许：多个文件名、多个目录名混用

输入信息包含：
- 现有需求归档与分析：`/specs/background/original.md`
- 现有细节规格：`/specs/details/`（必须存在且非空；否则本次 refine 不执行）
- 修订输入（优先级从高到低）：
  1. 命令参数指定的输入路径（文件/目录）
  2. 用户在对话窗口补充的“本次要修改的内容/变更点/新口径”（如有）
  3. `/docs/refine/refine.md`（如存在）
  4. `/docs/refine/`（可选补充材料：仅在 `refine.md` 或对话补充内容中引用/提及或明确需要时读取；不要无脑全量读取）
  5. `/specs/background/refine.md`（兼容旧路径）
  6. `/refine.md`（兼容旧路径）
- 如需核对上下游影响，可参考：`/specs/background/*.md`、`/specs/flows/*.puml`、`/specs/functions/*`

修订目标（必须）：
1. 将“修订指令”转化为一份清晰、可执行、无矛盾的“当前生效需求（Canonical Requirement）”
2. 保留可追溯性：输出必须包含“变更清单”（新增/修改/删除/澄清/优先级变化）
3. 不要覆盖或删除历史内容：只允许在 `original.md` 末尾追加一个新的“修订版本”段落

工作方式（必须）：
0. 执行前置条件（必须）：
   - 若 `/specs/details/` 不存在或为空：输出“无法执行：缺少 /specs/details（请先运行 /vspec:detail）”，并停止；不要写入或修改任何文件
   - 若未提供命令参数，且用户未在对话窗口提供修订内容，且 `/docs/refine/refine.md` 不存在：先提示用户在对话窗口粘贴本次需要修改的内容（可为变更点列表或完整新口径）；在用户提供后再继续本次 refine；在用户提供之前不要写入或修改任何文件
1. 解析修订输入（参数/默认 refine.md）：
   - 若提供了命令参数：逐个读取参数指向的内容作为修订输入
     - 若参数是目录：读取目录内所有可读文本文件（优先 `.md`，其次 `.txt`），按路径字母序合并为修订输入；忽略二进制与明显无关文件
     - 若参数是文件：直接读取该文件内容
   - 若未提供命令参数：
     - 若用户在对话窗口提供了修订内容：以该内容为主，并在需要时结合 `/docs/refine/refine.md`（如存在）作为补充材料
     - 否则以 `/docs/refine/refine.md` 为本次修订输入来源（必要时可再读取其引用/提及的补充材料）
   - 若输入中提供了完整的新需求文本：以其为主，并对照旧需求生成变更清单
   - 若输入只提供变更点/指令：基于旧需求生成修订后的 Canonical Requirement
1.1 外部附件与二进制材料读取（必须）：
   - 若 `refine.md`（或对话补充内容）中登记/引用了外部文件（包含但不限于：ppt/pptx、doc/docx、xls/xlsx、pdf，以及 png/jpg/jpeg/webp 等图片），必须读取这些文件并抽取可用信息，再将其合并进本次修订输入；禁止只看 `refine.md` 文本而忽略附件。
   - 文件定位方式（必须全部尝试）：
     - Markdown 链接：`[...](path/to/file.xxx)`、裸路径、相对路径与绝对路径
     - `/docs/refine/`、`/docs/current/`、`/docs/legacy/` 下的文件名引用（例如“见 xxx.pdf / xxx.pptx”）
     - `file_list.md` 中列出的路径（如存在且被引用）
   - 文本抽取要求（必须）：
     - PDF：按页抽取文字；若某页主要为图，必须对图做结构化描述（见“图与布局信息”）
     - PPT/PPTX：按页抽取标题、要点与备注；对流程图/架构图/布局图所在页做逐页摘要
     - Word（DOC/DOCX）：按标题层级抽取要点；对表格/字段口径/规则段落做结构化摘录
     - Excel（XLS/XLSX）：逐 Sheet 读取；识别“字段表/枚举/流程节点/状态流转/权限矩阵/角色映射”等表格并结构化总结；保留关键列名与口径
   - 图与布局信息利用（必须）：
     - 对文件中的流程图、状态图、布局图、线框图、架构图等，必须进行“可复述”的结构化描述（节点/连线/泳道/角色/状态/触发条件/异常分支/页面区域与控件/字段分组等）
     - 若图中存在关键文案/字段/状态/按钮/节点名称，必须提取并统一成术语（与 terms 尽量对齐）
     - 将图中信息显式映射回：主流程、关键规则、状态机/节点、权限与数据权限、页面交互与校验
   - 合并策略（必须）：
     - 将附件抽取内容先整理为“外部材料摘要（含来源文件路径）”，作为本次修订输入的一部分
     - 当附件与 `refine.md` 文本冲突：以“更明确、更晚更新、更权威（合同/规范/最终评审版）”的来源优先，并在变更清单标注冲突解决依据
   - 兜底（必须）：
     - 若某附件无法读取或信息不完整：不得跳过；必须在“待确认问题（修订新增）”中列出缺失项与所需补充（例如要求提供导出文本/截图/关键页）
2. 以“最后一份 Canonical Requirement”为基线：
   - 若 `original.md` 中存在“Canonical Requirement”标题（以下任意一种），以最后一次出现的该段内容作为基线：
     - 语言=en：`## Canonical Requirement`
     - 语言=zh-CN：`## 当前生效需求（Canonical）`
     - 语言=ja：`## 現行要件（Canonical）`
   - 否则，以“Raw Requirement/原始需求/要件原文”小节下的原文作为基线（只取原文，不取后续分析段落）：
     - 语言=en：`# Raw Requirement`
     - 语言=zh-CN：`# 原始需求`
     - 语言=ja：`# 要件原文`
3. 合并与消歧：
   - 若多个修订输入之间存在冲突：按命令参数顺序（或合并顺序）后者覆盖前者，并在变更清单标注“冲突解决（输入间）”
   - 若修订输入与基线冲突：优先采用修订输入，并在变更清单标注“冲突解决（覆盖基线）”
   - 若仍存在无法确定的地方，写入“待确认问题（修订新增）”

4. 影响分析与产物同步（必须）：
   - 基于“变更清单 + Canonical Requirement”，输出一份影响分析清单（包含：影响模块、影响功能点、影响的 detail 文档类型、是否影响原型页面/路由/组件）
   - 必须更新 `/specs/details/` 下受影响模块的文档（优先更新既有文件，避免无意义新文件）：
     - 若变更涉及角色/权限：更新 `rbac/` 与 `data_permission/`
     - 若变更涉及页面交互/字段/流程：更新 `interaction/`、`page_load/`、`validation_matrix/`、`post_submit_*` 等相关文档
    - 若变更涉及状态/操作可用性：更新 `state_machine/<module_slug>.*`（如存在）与 `decision_matrix/`、`validation_matrix/`
     - 若变更涉及外部依赖：更新 `dependencies.md` 对应模块的调用时机与失败兜底，并在细节规格中体现
   - 必须更新 `/specs/prototypes/`：
     - 页面/路由/菜单/按钮/表单字段必须与最新的细节规格一致
     - 仅做必要改动，保持最小可评审 diff
   - 若 `/specs/backend/` 存在且非空：必须同步更新受影响的后端代码，保证与最新 Canonical Requirement 与 `/specs/details/` 一致（仅做必要改动，保持最小可评审 diff）

Canonical Requirement 的表达要求（必须）：
- 用结构化的需求文本表达，避免口号式描述
- 至少包含：目标/范围（含不做什么）/角色与权限假设/主流程概览/关键规则与约束/关键数据口径（可概述，不到字段级）/外部依赖与边界（如有）
- 与已有产物保持可对齐：术语尽量沿用 terms；节点用 apply/approve/execute/change/cancel 等词汇
